# 04 — Readequação da Seção 1 (Hero) para 1366×768 / 1280×720 (laydesk2 / laydesk3)

## Objetivo

Readequar **a seção Hero** para que o layout fique fiel à **realidade do monitor** (ex.: **1366×768**) e não ao que foi visto durante desenvolvimento com **zoom do navegador em 50%**.

**Requisitos**
- **Não implementar nada ainda** (apenas documentação/plano).
- **Não mudar outros layouts** (mobile e laydesk1).
- Ajustar primeiramente **apenas a seção Hero** (Seção 1).
- Corrigir o **tamanho da seção** e impedir overflow/rolagem indesejada.

---

## Contexto: por que o DevTools “enganou”

Quando o site foi “ajustado” com **zoom do navegador em 50%**, o browser passou a “caber” mais conteúdo no mesmo monitor. Na prática, isso **muda o tamanho do viewport em CSS pixels**, então o layout parecia perfeito no DevTools, mas em **100%** (realidade do usuário) o conteúdo **cresce/estoura**.

### Check rápido (obrigatório antes de mexer no layout)

No monitor real onde vamos validar (ex.: 1366×768):
- **Windows Display Scale**: 100% (recomendado para calibração)
- **Zoom do navegador**: 100% (Ctrl+0)
- No console, conferir:
  - `window.innerWidth` / `window.innerHeight`
  - `window.devicePixelRatio`

**Critério:** só considerar “real” o teste quando **zoom = 100%** e você estiver olhando a página **fora do modo emulação**.

---

## Diagnóstico do Hero hoje (estrutura + risco atual)

### Estrutura do componente

O Hero (`components/hero/Hero.tsx`) tem 3 blocos verticais:
1. **Título (H1)** — `h1.laydesk3-hero-h1`
2. **Miolo** — grid 2 colunas (carousel + texto + diferenciais)
3. **Métricas** — grid 3 cards — `.laydesk3-hero-metrics`

### Problema atual observado

Dentro do media query do **laydesk3** existem regras “definitivas” para:
- `.laydesk3-hero-content-wrapper`
- `.laydesk3-hero-carousel`, `.laydesk3-hero-carousel-img`
- `.laydesk3-hero-feature-card`
- `.laydesk3-hero-metrics-grid`, `.laydesk3-hero-metric-*`

**Porém essas classes NÃO existem no JSX atual do Hero.**

Consequência:
- Parte do CSS **não tem efeito** (fica “no vácuo”).
- Ajustes viram tentativa/erro com seletores frágeis.

---

## Estratégia (plano) — “Recalibrar” o Hero de forma robusta

### Meta principal

Fazer o Hero caber **sem rolagem** em:
- **laydesk2 (1366×768)** em zoom 100%
- **laydesk3 (1280×720)** em zoom 100%

Sem depender de gambiarras como:
- `line-height: 0`
- `margin-top: -10vh`
- valores gigantes (`30rem`) para reposicionar elementos

### Princípio técnico

Em vez de “fixar” tudo com posicionamento absoluto, a abordagem mais estável é:
- Hero vira um **flex column**
- O miolo tem `min-height: 0` e limites por `max-height`
- As métricas ficam “presas embaixo” com `margin-top: auto` (sem `position:absolute`)

Isso evita overlap e mantém previsível em 768/720 de altura.

## Proposta de mudanças (para implementar depois) — em passos

> **Importante:** os passos abaixo descrevem “o que fazer” depois que você autorizar.  
> Nesta fase, **não implementar**.

### Passo 1 — Criar “ganchos” de CSS (classes) no JSX do Hero

Adicionar classes específicas (somente como hooks) em `components/hero/Hero.tsx`:
- **Wrapper do conteúdo do Hero** (logo abaixo do H1): `laydesk3-hero-content-wrapper`
- **Container do carousel**: `laydesk3-hero-carousel`
- **Imagem do slide** (ou wrapper do `Image`): `laydesk3-hero-carousel-img`
- **Cards diferenciais** (componente `FeatureItemCompact`): permitir className hook tipo `laydesk3-hero-feature-card`
- **Métricas**:
  - container: `laydesk3-hero-metrics`
  - grid interno: `laydesk3-hero-metrics-grid`
  - número/label: `laydesk3-hero-metric-number`, `laydesk3-hero-metric-label`

**Por que?** Para que o CSS do laydesk2/3 seja preciso e não dependa de `nth-child`.

### Passo 2 — Normalizar a altura da seção (evitar “tamanho errado”)

No `laydesk2` e `laydesk3`, padronizar:
- Hero com altura disponível abaixo do header fixo:
  - \(height = 100svh - 4rem\) (ou uma variável real de altura do header)

**Recomendação:** usar `svh` em vez de `vh` para reduzir bugs de viewport.

### Passo 3 — “Orçamento de altura” (distribuir espaço entre blocos)

Definir uma regra simples de orçamento:
- H1: compacto (tamanho e margens controlados por `clamp`)
- Miolo (carousel + texto + diferenciais): `max-height` relativo (ex.: `45svh`)
- Métricas: sempre visíveis (compactar padding e fontes se necessário)

**Evitar**: prender métricas com `position:absolute` — só usar se for inevitável.

### Passo 4 — Ajustes de escala por layout (laydesk2 vs laydesk3)

Criar dois perfis:
- **laydesk2 (768px altura)**: compactar moderadamente
- **laydesk3 (720px altura)**: compactar mais (reduzir gaps/paddings/typography)

**Preferir** `clamp()` + variáveis CSS (ex.: `--hero-gap`, `--hero-pad`) a números “hardcoded” demais.

## Proposta de CSS (exemplo — para colar depois, mediante autorização)

> Abaixo é um “esqueleto” (não é para aplicar agora).  
> A implementação final deve substituir regras frágeis e usar apenas hooks reais.

```css
/* laydesk2: 1366×768 (altura entre 768 e 899) */
@media (min-width: 1024px) and (min-height: 768px) and (max-height: 899px) {
  .laydesk3-hero-section {
    height: calc(100svh - 4rem) !important;
    overflow: hidden !important;
    display: flex !important;
    flex-direction: column !important;
  }

  h1.laydesk3-hero-h1 {
    font-size: clamp(1.4rem, 2.2vw, 2.2rem) !important;
    line-height: 1.1 !important;
    margin-bottom: 0.75rem !important;
  }

  .laydesk3-hero-content-wrapper {
    min-height: 0 !important;
    max-height: 48svh !important;
  }

  .laydesk3-hero-carousel {
    max-height: 34svh !important;
  }

  .laydesk3-hero-metrics {
    margin-top: auto !important; /* empurra para o rodapé sem absolute */
    padding-top: 0.5rem !important;
  }
}

/* laydesk3: 1280×720 (altura <= 767 no projeto atual) */
@media (min-width: 1024px) and (max-height: 767px) {
  .laydesk3-hero-content-wrapper { max-height: 44svh !important; }
  .laydesk3-hero-carousel { max-height: 30svh !important; }
  .laydesk3-hero-feature-card { padding: 0.35rem 0.5rem !important; }
  .laydesk3-hero-metric-number { font-size: 0.95rem !important; }
  .laydesk3-hero-metric-label { font-size: 0.6rem !important; }
}
```

## Checklist de validação (obrigatório)

Validar com **zoom 100%**:

### 1366×768 (laydesk2)
- [ ] Hero ocupa exatamente a área abaixo do header (sem “sobrar” e sem cortar)
- [ ] Sem scroll vertical “gratuito” dentro do Hero
- [ ] Carousel visível e setas clicáveis
- [ ] Texto e diferenciais não empurram as métricas para fora
- [ ] Métricas visíveis integralmente

### 1280×720 (laydesk3)
- [ ] Mesmos critérios do laydesk2
- [ ] Sem overlap de métricas sobre o conteúdo
- [ ] Tipografia legível (mesmo compacta)

### Não-regressão
- [ ] laydesk1 (≥1600×900) não muda
- [ ] mobile não muda
